# 餐馆取号排队网站设计说明

## 2026-07-13 意向图提取

参考文件：`design-reference/order-entry-concept.png`

### 设计判断

这是给餐厅前台员工使用的移动端点单工作台。设计优先级依次为操作速度、状态辨识、低误触和视觉秩序，不使用营销页式的大标题、数据看板或装饰图片。

- `DESIGN_VARIANCE: 4`：布局有桌面端主次分栏，但操作路径保持线性。
- `MOTION_INTENSITY: 3`：只保留按压反馈、步骤切换和成功确认。
- `VISUAL_DENSITY: 6`：一屏展示足够多的号码，同时保持至少 48px 的点击区域。
- 设计基础：原生 CSS 变量与系统中文黑体，不引入营销型组件库。

### 从意向图提取的结构

1. 顶部导航保持单行，当前入口用短紫色下划线标记。
2. 四步进度独占一行，用编号、连线和中文动作表达当前位置。
3. 桌面与 iPad 横屏采用主操作区加右侧订单摘要的分栏，号码网格约占三分之二以上宽度。
4. 号码是页面最大视觉对象。正常、已选和使用中三种状态同时通过文字、图标、底色和边框区分。
5. 底部动作区固定在可触达位置，次要清空操作与深色主按钮明确分级。
6. 小屏改为单列，订单摘要压缩成操作区上方的短摘要，流程顺序不变。

### 提取的视觉令牌

| 令牌 | 值 | 用途 |
| --- | --- | --- |
| `--canvas` | `#FFFFFF` | 页面与主要操作面 |
| `--ink` | `#292D34` | 号码、标题、主要文字 |
| `--slate` | `#646464` | 辅助文字 |
| `--mist` | `#F3F3F5` | 禁用态和次级表面 |
| `--line` | `#E3E3E7` | 分隔线与未选边框 |
| `--iris` | `#6F5BE7` | 当前步骤、选中态、焦点 |
| `--charcoal` | `#202020` | 最终操作按钮 |
| `--success` | `#178A55` | 下单成功与可取餐 |
| `--danger` | `#D14343` | 超时与危险操作 |

控件使用 10px 圆角，主要表面使用 16px 圆角，号码保持圆形。全站不使用投影、玻璃效果和大面积渐变。

## 一句话方向

这是一个给餐馆员工使用的移动端点单和出餐工具。它不追求“漂亮得像宣传页”，而要做到醒目、直接、少出错：员工拿起手机或 iPad，就能按“选号码 → 选品类 → 选小料 → 下单”的顺序快速完成操作。

整体感觉参考所给设计中的白底、深色文字、少量紫色提示和无阴影做法，但会改成更适合餐馆忙碌环境的高对比、大按钮界面。

## 1. 这个页面要表达什么

网站要让人一眼明白三件事：

- 现在正在为哪个号牌点单。
- 这位顾客点了什么、加了哪些小料。
- 哪些订单等待制作、正在制作或可以取餐。

点单页只服务一条主流程：

**选择号数 → 选择品类 → 选择小料（可多选）→ 确认下单**

完成一步后，下一步自然出现或自动获得重点。页面不放欢迎语、功能介绍、广告、经营数据等与当前点单无关的内容。

整个产品包含三个主要界面：

1. **点单界面**：员工选择号牌、品类和小料，然后下单。
2. **出餐界面**：后厨或出餐员工查看订单，按下单顺序或按品类处理。
3. **店内大屏**：展示等待中和可取餐的号码，只给顾客看进度，不展示小料和内部操作。

后台的菜品品类、小料和排序设置属于管理功能，入口放在次要位置，不干扰日常点单。

## 2. 页面给谁看

### 主要使用者

- 前台点单员工：需要单手快速选择，不能反复确认和输入文字。
- 后厨与出餐员工：需要隔着一段距离也能看清号码、品类和等待时间。
- 店长或管理员：偶尔编辑品类、小料，切换出餐排序方式。

### 大屏观看者

- 店内等待的顾客：只关心自己的号码是否已经可以取餐。

员工可能戴手套、手上有水、周围光线复杂，也可能同时处理多位顾客。因此所有常用按钮都要大，文字要少，选中状态必须非常明显。手机和 iPad 都使用同一套逻辑；iPad 只是同屏展示更多内容，不改变操作顺序。

## 3. 参考里要学什么

- **白底加深色文字**：画面干净，号码和订单内容更容易被看见。
- **只用一种强调色**：紫色只负责表示选中、当前步骤和主要操作，不到处装饰。
- **扁平、无阴影**：依靠底色、边框和留白区分区域，避免界面显得花哨。
- **紧凑而有秩序**：同类内容排得整齐，员工能快速扫过，而不是阅读大段说明。
- **圆角有统一规律**：小按钮用较小圆角，订单卡片用稍大圆角，不随意变化。
- **深色区域只放在真正重要的位置**：例如底部“确认下单”按钮、出餐页的重点状态，不让整页发黑。

这些做法只作为视觉语言，不照搬参考网站的页面布局、品牌标识、英文文案、产品图标或营销内容。

## 4. 参考里不要学什么

- 不要参考网站式的大标题、品牌故事、客户 Logo、宣传模块和页脚。
- 不要照搬参考品牌名称、英文文案、图标、插图和任何素材。
- 不要使用彩虹渐变、发光边缘或装饰性彩色图形。
- 不要把紫色铺成大面积背景；大面积高饱和颜色在忙碌环境里容易干扰判断。
- 不要为了“高级感”使用过多留白。操作区要紧凑，但按钮之间仍要有足够距离，避免误触。
- 不要用很小的灰字承载重要信息。号牌、品类、状态和等待时间必须高对比。
- 不要照搬桌面网页的横向导航。移动端只保留点单、出餐、设置三个必要入口。

## 5. 颜色怎么定

整套界面采用“白、深灰、紫色提示”三层颜色，不使用杂乱的品类彩色标签。

| 用途 | 颜色 | 怎么用 |
| --- | --- | --- |
| 页面底色 | `#FFFFFF` 白色 | 全站主要背景，保持干净明亮 |
| 次级底色 | `#F3F3F5` 浅灰 | 未选按钮、分组区、顶部步骤栏 |
| 主要文字 | `#292D34` 深灰 | 号码、品类、订单内容和标题 |
| 次要文字 | `#646464` 中灰 | 时间、数量、次要提示 |
| 边框 | `#E3E3E7` 淡灰 | 分隔卡片和未选项目 |
| 操作强调 | `#6F5BE7` 清晰紫色 | 当前步骤、选中状态、主要操作提示 |
| 深色按钮 | `#202020` 近黑 | “确认下单”等最终操作 |
| 完成状态 | `#178A55` 绿色 | 只用于“可取餐”“已完成” |
| 警示状态 | `#D14343` 红色 | 只用于超时、删除和错误，不作为装饰 |

具体规则：

- 正常界面约七成是白色和浅灰色，两成是深色文字，一成以内是紫色或状态色。
- 选中号牌使用紫色实心圆和白色数字，未选号牌使用白底、深色数字、浅灰边框。
- “可取餐”必须同时出现绿色和文字，不能只靠颜色判断。
- “等待中”“制作中”“可取餐”除了颜色不同，还要有明确中文标签。
- 大屏使用深灰背景配白色大号数字；“可取餐”区域可用绿色，确保在店内远处也能看清。
- 不用彩虹色，不给每个品类随意分配一种颜色，不使用大面积渐变。

## 6. 字体大概是什么感觉

字体要像清楚、可靠的工具，不像海报或潮牌。

- 中文优先使用手机自带的黑体：iPhone/iPad 用苹方，安卓用系统黑体。无需让商家额外下载字体。
- 字形要简洁、笔画清楚，数字“1、7、0、8”在远处也容易分辨。
- 号牌数字是页面最大、最粗的内容。
- 品类名和按钮文字使用中粗字；辅助信息使用正常字重。
- 全站最多使用四档字号，避免每块内容都不一样。

建议大小：

- 号牌圆圈：数字约 26–32 像素；大屏号码根据屏幕尽量放大。
- 页面标题和当前步骤：20–24 像素。
- 品类、规格、主要按钮：17–20 像素。
- 时间和辅助信息：14–16 像素，但不能小于 14 像素。

文案尽量短，例如“选号码”“选品类”“选小料”“确认下单”“开始制作”“通知取餐”。不要用绕弯的句子。

## 7. 首屏怎么排

### 点单首屏

进入后直接开始点单，不显示欢迎页。

从上到下这样排：

1. **顶部栏**：左侧显示当前功能“点单”，右侧只有“出餐”和更多设置入口。顶部栏保持固定。
2. **四步进度**：用短条显示“号码 / 品类 / 小料 / 确认”。当前步骤为紫色，完成的步骤带对勾。它只提示位置，不要求员工点进点出。
3. **号牌区**：首屏主体直接展示 1–100 号圆形号牌。手机每行 4–5 个，iPad 每行 7–10 个。常用号码可快速滚动，已占用号码变灰并标出“使用中”。
4. **当前订单条**：选中号码后，在底部按钮上方固定显示简短摘要，例如“28号 · 套餐A · 加辣、加葱”。
5. **底部主按钮**：固定在拇指容易点到的位置。未完成必要选择时显示下一步提示，如“请先选号码”；完成后变为深色“确认下单”。

选中号牌后，不跳出复杂弹窗，而是顺势进入品类区。品类使用大块按钮；选中品类后直接展示该品类对应的小料。若该品类没有小料，则直接进入确认状态，不多增加一步。

下单成功后，页面给出明确反馈“28号下单成功”，短暂停留后自动回到空白点单状态，准备接下一单。

### 出餐首屏

顶部先显示两个切换项：

- **按下单顺序**：默认方式，最早订单在最前。
- **按品类查看**：先按品类分组，每组内部仍按下单先后排列。

下方直接展示待处理订单。每张订单卡最醒目的是号牌数字，其次是品类和小料，右上角显示等待时长。卡片底部只放当前阶段最需要的按钮，例如“开始制作”或“可取餐”。

手机为单列；iPad 可为双列或三列，但顺序始终从左到右、从上到下，不能因屏幕变宽而打乱先后关系。

### 店内大屏首屏

大屏分成两块：

- 左侧或上方为“制作中”，使用白色大号数字。
- 右侧或下方为“请取餐”，使用绿色高亮和更大的数字。

页面不显示订单详情、员工按钮、后台入口和解释文案。号码变化时有轻微提示，让顾客注意到新状态。

## 8. 按钮和卡片怎么做

### 号牌按钮

- 必须是圆形，视觉上像真实号牌。
- 手机上的可点范围至少 48×48 像素，圆圈之间保留空隙。
- 未选：白底、浅灰边框、深色数字。
- 已选：紫色实心、白色数字，并有清楚的对勾或加粗边框。
- 使用中：浅灰底、灰字，并直接写“使用中”，不能点了才告诉员工。

### 品类按钮

- 使用大块矩形按钮，一屏尽量看到主要品类。
- 白底、浅灰边框、约 10 像素圆角。
- 选中后使用浅紫底、紫色边框和深色文字，不只改变一条细边。
- 品类名称保持一到两行；过长名称在后台提示管理员缩短，不在前台挤成小字。

### 小料多选按钮

- 使用宽大的方圆角选择块，不用细小复选框。
- 每个按钮直接显示小料名；选中后出现勾号。
- 如果有加价，价格放在名称下方或右侧，例如“加蛋 +2元”，但不让价格抢过小料名。
- 已选数量和内容在底部订单摘要中同步更新。

### 主要按钮

- “确认下单”“可取餐”这类主操作占满底部主要宽度，高度不少于 52 像素。
- 每个画面只保留一个最醒目的主按钮。
- 最终下单按钮使用近黑底和白字；状态操作可以使用对应的绿色。
- 删除、撤销等危险操作不与主按钮并排抢位置，要二次确认并明确写出影响的号牌。

### 订单卡片

- 白底、1 像素浅灰边框、约 16–18 像素圆角，不加投影。
- 左侧用大号圆形号码作为视觉中心。
- 中间从上到下显示品类、小料、备注；没有备注时不保留空行。
- 右侧或右上角显示等待时间。超过店家设定时间后才变红。
- 卡片底部操作区域与内容区有明显分隔，避免点错订单。
- 已完成订单从主列表移走，进入可查看但不打扰当前工作的记录区。

### 后台编辑卡片

- 一个品类对应一个简单卡片，可编辑名称、排序、启用状态和可选小料。
- 小料默认支持多选；管理员可以设置必选、最多可选数量和是否加价。
- 删除品类前必须提示它会影响哪些正在使用的设置；已产生的历史订单不随后台修改而改变。

## 9. 动效做到什么程度

动效只负责确认状态，不负责表演。

- 点击号牌、品类、小料时，在 0.15–0.2 秒内完成颜色和边框变化。
- 步骤切换使用轻微的淡入或上下移动，不整页横飞，不做旋转和弹跳。
- 下单成功时，按钮短暂变成“下单成功”，配一个对勾；约 1 秒后回到新订单状态。
- 订单进入“可取餐”时，卡片轻微高亮一次；店内大屏的新号码可闪亮或轻微放大一次，但不持续闪烁。
- 删除、失败、网络中断要有明确静止提示，不能只靠震动或一闪而过的消息。
- 网络慢时保留员工刚才的选择，按钮显示“正在下单”，避免重复提交。
- 不使用彩带、粒子、玻璃模糊、漂浮光斑和长时间渐变动画。

所有动效以不拖慢连续点单为前提。员工关闭系统动效后，界面仍要完整可用。

## 10. 哪些常见 AI 味不要出现

- 不要大面积紫蓝渐变、发光光圈、玻璃拟态和悬浮阴影。
- 不要把所有卡片都做成巨大圆角的“胶囊”；圆形只留给号牌。
- 不要无意义的机器人、魔法星星、闪电、火箭和抽象 3D 图标。
- 不要每个品类配一张风格不一致的 AI 菜品图。这个系统靠文字和号码快速识别，不依赖图片。
- 不要出现“智能赋能”“高效协同”“开启美味之旅”等空泛文案。
- 不要为了显得丰富加入经营数据、天气、问候语、轮播图和快捷入口宫格。
- 不要用大量英文小标题、全大写标签或难懂的后台术语。
- 不要让每张卡片都拥有不同颜色，也不要用颜色代替清楚的状态文字。
- 不要设计一眼看着漂亮、实际却需要连续打开弹窗的流程。
- 不要在忙碌操作中播放夸张庆祝动画或要求员工等待动画结束。
- 不要生成虚构的品牌 Logo、店名、宣传口号或顾客评价。

## 最后检查标准

设计稿完成后，用下面几条判断是否合格：

- 员工进入页面后，是否不用阅读说明就能开始选号码？
- 单手操作时，常用按钮是否足够大、够近、不会误触？
- 在嘈杂和匆忙的环境里，是否一眼能看清当前号牌和下一步？
- 选中、禁用、等待、制作中、可取餐，是否不只靠颜色也能分辨？
- 手机和 iPad 上是否保持同一操作顺序？
- 出餐员工是否能在三秒内找到最早的待处理订单？
- 顾客能否从店内远处看清自己的取餐号码？
- 页面里是否删除了所有与当下任务无关的说明和装饰？

如果以上答案都是“是”，这套设计才算真正适合餐馆使用。
