# 号牌账单、后厨拆单、顾客进度与结算需求

## 背景与范围

界面继续使用商家现有的“号牌”称呼，但业务模型从“一张号牌只能对应一道菜”升级为“一张号牌对应一张未结算账单”。同一号牌可以持续加入多道菜，后厨按菜品明细独立制作，前台在全部菜品完成后统一结算。

本次变更还包括菜品大类编辑、号牌二维码顾客进度页、商家收款码、结算界面以及与新账单模型一致的经营统计和数据导出。项目尚未上线，不迁移当前测试数据库；实现时直接重建测试数据。

## 用户故事

1. 作为商家，我希望能修改已经创建的菜品大类名称，避免因录入错误只能逐道修改菜品。
2. 作为点菜人员，我希望选择号牌后一次添加一道菜，并能在顾客继续加菜时再次进入同一号牌下单。
3. 作为点菜人员，我希望在号牌列表中一眼区分空闲号牌和已有未结算账单的号牌。
4. 作为点菜人员，我希望随时看到该号牌当前账单的菜品明细和累计消费金额。
5. 作为后厨人员，我希望每一条菜品明细成为独立制作任务，并且只需点击“开始制作”和“完成制作”两次操作。
6. 作为顾客，我希望扫描号牌背面的二维码查看每道菜的制作状态、排队进度、价格和账单总额。
7. 作为顾客，我希望在进度页底部看到商家的收款码，方便自行付款。
8. 作为收银人员，我希望在结算界面查看每个号牌的总消费、制作进度和结算状态，并在全部菜品完成后完成结算。
9. 作为经营者，我希望营业额、订单历史和导出数据以已结算账单为准，同时保留每道菜的完整制作时间。

## 术语与核心规则

- **号牌**：沿用现有可自定义、可不连续的号码列表；所有商家端和顾客端界面均继续显示“号牌”，不改称“桌号”。
- **号牌账单**：内部采用类似桌台账单的逻辑。同一账号下，一个号牌同一时间最多存在一张未结算账单。
- **菜品任务**：每次向号牌账单加入的一条菜品明细。任务保存菜品、品类、价格、小料和份数快照。
- **未结算账单**：允许继续加菜；即使已有菜品完成制作，也不自动关闭账单。
- **已结算账单**：禁止继续加菜，进入经营历史，并释放号牌以便创建下一张账单。
- 菜品任务由系统创建时自动进入“待制作”，后厨不需要点击确认待制作。
- 后厨对一条菜品任务只操作两次：第一次“开始制作”，第二次“完成制作”。
- 同一次下单选择多份相同菜品时，形成一条带份数的菜品任务；之后再次加同一道菜时，形成新的菜品任务，以保留各自的下单和制作时间。
- 只有账单内所有菜品任务均已完成制作时，系统才允许结算。
- 每个号牌拥有一个长期有效、可打印的专属二维码；二维码关联号牌身份，而不是某一张具体账单。
- 本期结算仅记录结算状态与结算时间，不处理支付渠道、折扣、退款、抹零、分单或部分结算。

## 验收标准

### 1. 菜品大类编辑

- 当商家打开菜品管理时，系统应显示当前账号的菜品大类，并为每个大类提供编辑名称的入口。
- 当商家提交新的大类名称时，系统应校验名称非空且不得与当前账号的其他大类重名。
- 当大类名称修改成功时，系统应同步更新该大类下所有现有菜品在点菜页和管理页中的分类显示。
- 当大类名称修改成功时，系统不得改写已经生成的菜品任务或历史账单中的品类快照。
- 当同一账号在其他设备查看菜单时，系统应实时同步大类名称变更。
- 不同账号之间的菜品大类及其编辑结果必须保持隔离。

### 2. 号牌与持续加菜

- 当用户查看点菜页或工作台设置时，系统应继续使用“号牌”文案。
- 当号牌没有未结算账单时，用户选择该号牌并提交第一道菜，系统应自动创建该号牌的未结算账单并加入菜品任务。
- 当号牌已有未结算账单时，系统应允许用户再次选择该号牌并继续加入菜品任务，不得锁定或禁用该号牌。
- 当点菜页显示号牌列表时，系统应通过清晰但不影响点击的视觉状态区分空闲号牌与已有未结算账单的号牌。
- 当号牌已有未结算账单时，号牌项应至少显示“进行中”状态和当前累计金额；空闲号牌不得显示该状态。
- 当用户重新进入已有账单的号牌时，系统应显示该号牌当前菜品明细、份数、单项金额和累计消费金额。
- 当用户加入菜品时，系统应沿用“品类 → 菜品 → 小料与份数 → 确认”的自动推进流程。
- 当新菜品加入成功时，系统应立即更新该号牌账单总额，并在同一账号的其他设备及顾客进度页实时显示。
- 当账单未结算时，系统不得因为其中所有菜品已经完成制作而自动释放号牌。
- 当商家尝试从设置中移除存在未结算账单的号牌时，系统应拒绝该操作并说明需要先完成结算。

### 3. 金额与账单计算

- 当菜品任务加入账单时，系统应按“（菜品基础价 + 所选小料单价之和）× 份数”计算该任务金额。
- 当账单包含多条菜品任务时，系统应以所有任务金额之和作为该号牌当前消费总额。
- 当菜品或小料之后被改价、停用、改名或删除时，系统不得改变已加入账单的价格与名称快照。
- 当用户查看点菜页、顾客进度页、结算页或账单详情时，系统应显示一致的号牌总消费金额。
- 当账单完成结算时，系统应保存最终账单金额，不得使用之后变更的菜单价格重新计算历史收入。

### 4. 后厨按菜品拆单

- 当任意号牌加入一道菜时，系统应立即为该菜品明细创建一条状态为“待制作”的后厨任务。
- 当一张号牌账单包含多道不同菜品时，系统应在后厨队列中分别显示对应菜品任务，不得将整张账单合并为一张制作卡片。
- 当后厨查看队列时，系统应以菜品名称、份数和小料信息为主要内容，同时保留号牌用于送餐识别。
- 当后厨选择“全部”筛选时，系统应按菜品任务的下单时间显示所有未完成任务。
- 当后厨选择某个菜品筛选时，系统应只显示该菜品的未完成任务，并显示该菜品待处理的任务数和总份数。
- 当后厨点击“开始制作”时，系统应将该菜品任务从“待制作”变为“制作中”并记录开始制作时间。
- 当后厨点击“完成制作”时，系统应将该菜品任务从“制作中”变为“已完成”并记录完成制作时间。
- 当菜品任务处于“待制作”时，界面应只提供“开始制作”；当其处于“制作中”时，界面应只提供“完成制作”。
- 系统不得要求后厨对同一菜品任务执行第三次确认，也不得提供“完成取餐”等额外后厨步骤。
- 当同一账号在多台设备登录时，新菜品任务及其状态变化应实时同步；不同账号之间不得互相看到后厨任务。

### 5. 号牌二维码与顾客进度页

- 当商家管理号牌时，系统应为每个号牌生成独立且长期稳定的二维码，并支持查看和下载以便打印粘贴。
- 号牌二维码应使用不可猜测的公开令牌定位号牌，不得直接暴露账号 ID、数据库主键、登录令牌或其他商家数据。
- 当顾客扫描二维码时，系统应打开无需登录的移动端只读页面，不得允许顾客新增菜品、修改任务或执行结算。
- 当该号牌存在未结算账单时，顾客页应显示号牌、全部菜品、份数、小料、单项金额、账单总额和整体完成进度。
- 当菜品任务处于待制作时，顾客页应显示其当前排队位置或前方待制作数量；当任务开始制作或完成制作时，应分别显示“制作中”或“已完成”。
- 排队位置应基于同一菜品的待制作队列和下单时间计算，且不得把其他商家的任务计入或暴露给顾客。
- 当后厨任务状态变化或号牌继续加菜时，顾客页应自动获得最新状态，无需手动刷新。
- 当号牌当前没有未结算账单时，顾客页应显示“暂无进行中的订单”，不得展示上一位顾客的历史账单明细。
- 当商家配置收款码时，顾客页底部应显示该收款码；当未配置时，页面不得显示空白图片或失效占位。
- 商家应能在“我的”中上传、替换或移除收款码，收款码数据必须与当前账号隔离。

### 6. 结算界面与号牌释放

- 主导航应包含四个标签：“点菜”“出餐”“结算”“我的”。
- 当用户进入结算界面时，系统应显示当前未结算号牌、账单总额、菜品数量、制作进度和结算状态。
- 当用户打开某号牌账单详情时，系统应显示全部菜品任务、份数、小料、单项金额、下单时间和制作状态。
- 当账单中仍有“待制作”或“制作中”的菜品任务时，系统应禁用结算操作并明确显示尚未完成的任务数量。
- 当账单中所有菜品任务均为“已完成”时，系统应允许用户点击一次“确认结算”。
- 当用户确认结算时，系统应记录精确到秒的结算时间，将账单状态变为“已结算”，并从未结算列表移至已结算记录。
- 当账单结算成功时，系统应释放该号牌；用户之后在同一号牌下单时，系统应创建一张新的独立账单。
- 当结算请求因重复点击或多设备同时操作而重复到达时，系统应只结算一次，不得重复生成历史记录或重复累计营业额。
- 当用户查看结算记录时，系统应能区分同一号牌在不同时间产生的多张账单，并显示每张账单是否已结算及其总消费。

### 7. 历史数据、统计与导出

- 当号牌账单完成结算时，系统应将整张账单及其全部菜品任务保存为经营历史，不得在单道菜完成制作时提前计为一张已完成订单。
- 当用户查看营业额、完成订单数和平均客单时，系统应只统计所选时间范围内已结算的号牌账单，并以结算时间作为归属时间。
- 当用户查看菜品销售统计时，系统应按已结算账单内的菜品任务及份数汇总销量。
- 当用户导出 CSV 或 JSON 时，系统应按结算时间筛选，并包含账单 ID、号牌、账单状态、总金额、开单时间、结算时间以及全部菜品任务明细。
- 每条导出的菜品任务应包含任务 ID、菜品与品类快照、份数、价格、小料、任务金额、下单时间、开始制作时间和完成制作时间。
- 导出的全部金额字段应以人民币“元”为单位，保留两位小数并明确标注币种为 CNY，不得要求使用者自行将“分”换算为“元”。
- 导出时间字段应同时保留原始 ISO 时间和北京时间到秒，JSON 应保留账单与菜品任务的层级关系，便于后续 AI 分析。
- 导出接口只能返回当前登录账号的数据。

### 8. 测试数据库重建与数据安全

- 实现本需求时，系统应直接重建当前测试数据库，不执行旧订单、旧历史或旧账号的数据迁移。
- 重建后的数据库应只预置固定测试账号 `yue`，密码为 `123`，并保留一套可直接体验点菜、后厨和结算流程的测试菜单、小料及号牌配置。
- 重建时应清除除 `yue` 以外的现有测试账号、会话、账单、历史和其他测试数据。
- `yue` 的弱测试密码只允许由初始化脚本创建，普通用户注册仍应遵守现有密码强度规则。
- 数据库重建不得被设置为每次服务启动时自动执行，避免未来上线后误删正式数据。
- 数据库重建后应继续开放普通用户注册，并为新账号创建相互隔离的独立菜单、号牌、账单、二维码令牌和经营数据。
- 所有登录接口、实时同步、顾客公开查询和导出接口均不得泄露其他账号的数据。

## 约束与非目标

- 保留“号牌”这一对外名称，只改变背后的账单关系。
- 主导航固定为四项：“点菜”“出餐”“结算”“我的”。
- 数据库存储仍使用整数“分”避免计算误差；界面和导出内容以人民币“元”展示。
- 号牌仍支持非连续号码配置，不强制从 1 到 N。
- 顾客二维码页面仅提供进度、价格和收款码展示，不提供在线支付能力。
- 本期不增加桌型、人数、预约、并桌、转桌、拆桌、挂账、优惠、退款、支付渠道或打印小票。
- 本期不支持部分菜品结算、分单结算或菜品未完成时提前结算。
- 本期不改变开放注册和账号数据隔离规则。
