# 桌号账单、后厨拆单与结账需求

## 背景与范围

当前系统以“一个号牌对应一道菜”为核心：号牌下单后会被占用，同一号牌不能继续加菜；后厨完成该菜后订单立即归档。新的经营方式需要改为“一个桌号对应一张未结账账单”，一张账单可以持续加入多道菜，后厨按菜品明细独立制作，前台在全部菜品完成后统一结账。

本次变更同时补齐菜品大类的编辑能力，并让桌号账单、后厨任务、结账记录、经营统计和数据导出保持一致。

## 用户故事

1. 作为商家，我希望能修改已经创建的菜品大类名称，避免因录入错误只能逐道修改菜品。
2. 作为点菜人员，我希望选择桌号后一次添加一道菜，并能在顾客继续加菜时再次进入同一桌号下单。
3. 作为点菜人员，我希望随时看到该桌当前账单的菜品明细和累计消费金额。
4. 作为后厨人员，我希望每一条菜品明细成为独立制作任务，并且只需点击“开始制作”和“完成制作”两次操作。
5. 作为后厨人员，我希望按菜品筛选或汇总待制作任务，只关注需要制作的菜、份数和关联桌号。
6. 作为收银人员，我希望在结账界面查看每桌消费、制作进度和结算状态，并在全部菜品完成后完成结账。
7. 作为经营者，我希望营业额、订单历史和导出数据以已结账账单为准，同时保留每道菜的完整制作时间。

## 术语与核心规则

- **桌号**：沿用现有可自定义、可不连续的号码列表，但界面文案统一由“号牌/号码”调整为“桌号”。
- **桌台账单**：同一账号下，一个桌号同一时间最多存在一张未结账账单。
- **菜品任务**：每次向桌台账单加入的一条菜品明细。菜品任务保存菜品、品类、价格、小料和份数快照。
- **未结账账单**：允许继续加菜；即使已有菜品完成制作，也不自动关闭账单。
- **已结账账单**：禁止继续加菜，进入经营历史，并释放桌号以便创建下一张账单。
- 菜品任务由系统创建时自动进入“待制作”，后厨不需要点击确认待制作。
- 后厨对一条菜品任务只操作两次：第一次“开始制作”，第二次“完成制作”。
- 同一次下单选择多份相同菜品时，形成一条带份数的菜品任务；之后再次加同一道菜时，形成新的菜品任务，以保留各自的下单和制作时间。
- 只有账单内所有菜品任务均已完成制作时，系统才允许结账。
- 本期结账仅记录结算状态与结算时间，不处理支付渠道、折扣、退款、抹零、分单或部分结账。

## 验收标准

### 1. 菜品大类编辑

- 当商家打开菜品管理时，系统应显示当前账号的菜品大类，并为每个大类提供编辑名称的入口。
- 当商家提交新的大类名称时，系统应校验名称非空且不得与当前账号的其他大类重名。
- 当大类名称修改成功时，系统应同步更新该大类下所有现有菜品在点菜页和管理页中的分类显示。
- 当大类名称修改成功时，系统不得改写已经生成的菜品任务或历史账单中的品类快照。
- 当同一账号在其他设备查看菜单时，系统应实时同步大类名称变更。
- 不同账号之间的菜品大类及其编辑结果必须保持隔离。

### 2. 桌号与开台账单

- 当用户查看点菜页或工作台设置时，系统应使用“桌号”文案替代“号牌”和“号码牌”。
- 当桌号没有未结账账单时，用户选择该桌号并提交第一道菜，系统应自动创建该桌的未结账账单并加入菜品任务。
- 当桌号已有未结账账单时，系统应允许用户再次选择该桌号并继续加入菜品任务，不得因桌号正在使用而锁定或禁用。
- 当用户重新进入已有账单的桌号时，系统应显示该桌当前菜品明细、份数、单项金额和累计消费金额。
- 当用户加入菜品时，系统应沿用“品类 → 菜品 → 小料与份数 → 确认”的自动推进流程。
- 当新菜品加入成功时，系统应立即更新该桌账单总额，并在同一账号的其他设备实时显示。
- 当账单未结账时，系统不得因为其中所有菜品已经完成制作而自动释放桌号。
- 当商家尝试从设置中移除存在未结账账单的桌号时，系统应拒绝该操作并说明需要先完成结账。

### 3. 金额与账单计算

- 当菜品任务加入账单时，系统应按“（菜品基础价 + 所选小料单价之和）× 份数”计算该任务金额。
- 当账单包含多条菜品任务时，系统应以所有任务金额之和作为该桌当前消费总额。
- 当菜品或小料之后被改价、停用、改名或删除时，系统不得改变已加入账单的价格与名称快照。
- 当用户查看点菜页、结账页或账单详情时，系统应显示一致的桌号总消费金额。
- 当账单完成结账时，系统应保存最终账单金额，不得使用之后变更的菜单价格重新计算历史收入。

### 4. 后厨按菜品拆单

- 当任意桌号加入一道菜时，系统应立即为该菜品明细创建一条状态为“待制作”的后厨任务。
- 当一张桌台账单包含多道不同菜品时，系统应在后厨队列中分别显示对应菜品任务，不得将整桌账单合并为一张制作卡片。
- 当后厨查看队列时，系统应以菜品名称、份数和小料信息为主要内容，同时保留桌号用于送餐识别。
- 当后厨选择“全部”筛选时，系统应按菜品任务的下单时间显示所有未完成任务。
- 当后厨选择某个菜品筛选时，系统应只显示该菜品的未完成任务，并显示该菜品待处理的任务数或份数。
- 当后厨点击“开始制作”时，系统应将该菜品任务从“待制作”变为“制作中”并记录开始制作时间。
- 当后厨点击“完成制作”时，系统应将该菜品任务从“制作中”变为“已完成”并记录完成制作时间。
- 当菜品任务处于“待制作”时，界面应只提供“开始制作”；当其处于“制作中”时，界面应只提供“完成制作”。
- 系统不得要求后厨对同一菜品任务执行第三次确认，也不得提供“完成取餐”等额外后厨步骤。
- 当同一账号在多台设备登录时，新菜品任务及其状态变化应实时同步；不同账号之间不得互相看到后厨任务。

### 5. 结账界面与桌号释放

- 当用户进入结账界面时，系统应显示当前未结账桌号、账单总额、菜品数量、制作进度和结算状态。
- 结账入口应在不增加第四个主导航标签的前提下，从点菜业务区域清晰可达。
- 当用户打开某桌账单详情时，系统应显示全部菜品任务、份数、小料、单项金额、下单时间和制作状态。
- 当账单中仍有“待制作”或“制作中”的菜品任务时，系统应禁用结账操作并明确显示尚未完成的任务数量。
- 当账单中所有菜品任务均为“已完成”时，系统应允许用户点击一次“确认结账”。
- 当用户确认结账时，系统应记录精确到秒的结账时间，将账单状态变为“已结账”，并从未结账列表移至已结账记录。
- 当账单结账成功时，系统应释放该桌号；用户之后在同一桌号下单时，系统应创建一张新的独立账单。
- 当结账请求因重复点击或多设备同时操作而重复到达时，系统应只结算一次，不得重复生成历史记录或重复累计营业额。
- 当用户查看结账记录时，系统应能区分同一桌号在不同时间产生的多张账单，并显示每张账单是否已结算及其总消费。

### 6. 历史数据、统计与导出

- 当桌台账单完成结账时，系统应将整张账单及其全部菜品任务保存为经营历史，不得在单道菜完成制作时提前计为一张已完成订单。
- 当用户查看营业额、完成订单数和平均客单时，系统应只统计所选时间范围内已结账的桌台账单，并以结账时间作为归属时间。
- 当用户查看菜品销售统计时，系统应按已结账账单内的菜品任务及份数汇总销量。
- 当用户导出 CSV 或 JSON 时，系统应按结账时间筛选，并包含账单 ID、桌号、账单状态、总金额、开台时间、结账时间以及全部菜品任务明细。
- 每条导出的菜品任务应包含任务 ID、菜品与品类快照、份数、价格、小料、任务金额、下单时间、开始制作时间和完成制作时间。
- 导出时间字段应同时保留原始 ISO 时间和北京时间到秒，JSON 应保留账单与菜品任务的层级关系，便于后续 AI 分析。
- 导出接口只能返回当前登录账号的数据。

### 7. 兼容迁移与数据安全

- 当现有数据库首次由新版本启动时，系统应在不删除账号、菜单、设置、排队数据和历史数据的前提下完成兼容迁移。
- 当旧数据中存在未完成的单菜订单时，系统应按账号和桌号转换为未结账桌台账单，并将每个旧订单转换为菜品任务。
- 当旧数据中存在已完成历史订单时，系统应保留原始金额和时间快照，并转换为可查询、可统计、可导出的已结账历史账单。
- 当迁移遇到旧记录缺失开始制作时间时，系统应保留空值，不得伪造时间。
- 迁移和新接口必须继续遵守账号级数据隔离，同一账号多设备实时同步不得退化。

## 约束与非目标

- 保留现有三个主导航标签：“点菜”“出餐”“我的”。
- 继续使用整数“分”保存所有价格，界面以人民币“元”展示。
- 桌号仍支持非连续号码配置，不强制从 1 到 N。
- 本期不增加桌型、人数、预约、并桌、转桌、拆桌、挂账、优惠、退款、支付渠道或打印小票。
- 本期不支持部分菜品结账、分单结账或菜品未完成时提前结账。
- 本期不改变开放注册和账号数据隔离规则。
