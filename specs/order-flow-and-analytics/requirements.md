# 点单流程与经营分析需求

## 范围

本次变更将工作台主导航收敛为“点菜、出餐、我的”，建立菜品与公共小料库的可选关系，并把完成取餐的订单持久化为历史数据，用于按时间范围统计经营表现。

## 用户故事

1. 作为点单人员，我希望按号码、品类、菜品、小料的顺序快速下单，减少寻找菜品的时间。
2. 作为商家，我希望维护一份公共小料库，并为每个菜品选择允许搭配的小料，避免重复录入。
3. 作为商家，我希望在“我的”中管理菜品、小料和工作台设置，保持主导航精简。
4. 作为经营者，我希望按历史时间查看营业额、订单量、客单价和热销排行。

## 验收标准

### 导航与点单

- 当用户登录工作台时，系统应只显示“点菜、出餐、我的”三个主导航标签。
- 当用户选择号码时，系统应自动进入菜品选择区域。
- 当用户选择品类时，系统应在同一页面下方显示该品类的启用菜品。
- 当用户选择菜品时，系统应进入小料选择界面，并只显示该菜品允许且已启用的小料。
- 当用户确认小料后，系统应直接创建订单并返回号码选择界面。
- 当菜品没有可选小料时，系统仍应允许直接下单。

### 菜品与小料库

- 当商家新增或编辑菜品时，系统应保存品类、名称、说明、基础价格、启用状态和允许的小料 ID。
- 当商家维护小料时，系统应从公共小料库保存名称、单独价格和启用状态。
- 当小料被删除时，系统应从所有菜品的允许小料关系中移除该小料。
- 当旧菜品没有小料关系字段时，系统应兼容迁移为允许当前全部小料，避免改变既有点单能力。

### 历史数据与分析

- 当订单完成取餐时，系统应记录完成时间并移入历史订单，不再彻底删除。
- 当用户查看数据看板时，系统应支持今天、近 7 天、近 30 天和自定义日期范围。
- 当选择时间范围时，系统应按完成取餐时间统计营业额、订单数、客单价、品类销量、菜品销量和小料销量。
- 当时间范围内没有订单时，系统应展示明确的空状态，所有汇总值为零。
- 历史订单不得进入实时出餐队列或号码牌占用判断。

## 约束

- 保留现有账号、菜单、设置和未完成订单。
- 价格继续使用整数分存储。
- 分析数据仅来自完成取餐的订单，不包含等待制作或制作中的订单。
- 本期不包含退款、折扣、支付渠道和多门店拆分。
